# workload-portability

## ADDED Requirements

### Requirement: データアクセス層の分離
データストアへのアクセスは単一のデータアクセス層（リポジトリ層）に集約されなければならない (MUST)。UI・ハンドラ・ビジネスロジックからデータストアの SDK やクライアントを直接呼び出してはならない (MUST NOT)。

#### Scenario: データストア移行時の影響範囲
- **WHEN** データストアを移行する（例: Supabase から Aurora へ）とき
- **THEN** 変更はデータアクセス層と接続設定に限定され、ビジネスロジックと UI のコードは変更不要である

### Requirement: 認証の抽象化
認証・セッション確認は、標準プロトコル（OIDC / JWT 等)または自前の薄い抽象化層を介して行われなければならない (MUST)。アプリケーションコードが特定の認証プロバイダの SDK 型に直接依存してはならない (MUST NOT)。

#### Scenario: 認証プロバイダの切り替え
- **WHEN** 認証基盤を移行する（例: Supabase Auth から Cognito へ）とき
- **THEN** 変更は抽象化層の実装に限定され、画面・API のコードは変更不要である

### Requirement: コンテナ化可能なビルド
アプリケーションは、フェーズ0 の時点から OCI コンテナイメージとしてビルド可能でなければならない (MUST)。デプロイ先が PaaS / FaaS であっても、同一のコードベースからコンテナイメージを生成する CI ジョブが維持されなければならない (MUST)。

#### Scenario: 実行基盤の移行
- **WHEN** 実行基盤を移行する（例: Vercel から ECS Fargate / EKS へ）とき
- **THEN** 既存のコンテナイメージビルドがそのまま利用でき、アプリケーションの書き直しは発生しない

#### Scenario: コンテナビルドの継続的検証
- **WHEN** CI パイプラインが実行されたとき
- **THEN** コンテナイメージのビルドと起動確認が成功する（デプロイ先が PaaS であっても検証は行われる）

### Requirement: Postgres 互換のプライマリデータストア
プライマリのリレーショナルデータストアは PostgreSQL 互換でなければならない (MUST)。プラットフォーム固有の NoSQL（Firestore 等）をプライマリデータストアとしてはならない (MUST NOT)。

#### Scenario: 標準ツールによるデータ移行
- **WHEN** データベースを別のホスティング先へ移行するとき
- **THEN** `pg_dump` / `pg_restore` 等の標準ツールでスキーマとデータを移行でき、SQL・スキーマの書き直しは発生しない

### Requirement: スキーマとプラットフォーム設定のコード管理
データベーススキーマはマイグレーションファイルとして、PaaS・クラウドの設定は設定ファイルまたは IaC としてバージョン管理されなければならない (MUST)。ダッシュボードでの手動設定のみに存在する構成があってはならない (MUST NOT)。

#### Scenario: 移行先での環境再現
- **WHEN** 新しいプラットフォーム上に環境を構築するとき
- **THEN** バージョン管理されたマイグレーションと設定の適用のみで、手動手順なしに同等の環境が再現される

### Requirement: プラットフォーム固有機能の依存管理
PaaS 固有の付加機能（Cloudflare Workers KV / D1 / Durable Objects、Vercel KV 等）を採用する場合、その依存箇所と移行時の代替手段が記録されなければならない (MUST)。

#### Scenario: 固有機能の棚卸し
- **WHEN** フェーズ移行の計画を開始したとき
- **THEN** プラットフォーム固有機能への依存一覧と各々の代替手段が参照でき、移行工数を見積もれる
