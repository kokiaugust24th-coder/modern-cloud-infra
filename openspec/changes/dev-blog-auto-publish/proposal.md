# Change: dev-blog-auto-publish

## Why

開発中に蓄積される情報（コミット履歴、OpenSpec の変更提案・設計判断、実装で得た学び）がリポジトリ内に留まり、外部への発信・知識共有に活かされていない。手動でのブログ執筆は執筆コストが高く継続しないため、「収集 → 下書き生成 → 安全性検査 → 人間承認 → 公開」を自動化するパイプラインを整備し、開発ログを継続的にブログとして発信できるようにする。

## What Changes

- Git 履歴・OpenSpec チェンジ（進行中/アーカイブ）・タスク進捗などの開発情報を期間指定で収集し、構造化された「開発ダイジェスト」として出力する仕組みを追加する
- 開発ダイジェストからブログ記事の下書き（Markdown + frontmatter）を LLM 支援で生成する仕組みを追加する
- 記事品質を担保する仕組みを追加する: 版管理された記事テンプレート・スタイルガイド・品質ルーブリックに基づく多段生成（構成案 → 本文 → 自己批評リビジョン）、mermaid 図解と情報源リンクカードの自動付与、機械検証可能な記事リンター（frontmatter・見出し構造・文字数・リンク到達性・日本語校正）
- 開発者が日々の学び・参照 URL・埋め込み候補（X ポスト等）を書き溜める開発メモファイルを収集対象に加え、記事の素材を人手の負担なく増やせるようにする
- 記事に機密情報（API キー、認証情報、内部 URL、個人情報）が混入していないことを検査するサニタイズゲートを追加する
- 生成された下書きを人間がレビュー・承認（Pull Request ベース）した上でブログへ公開するワークフローを追加する
- 公開履歴（どの情報源からどの記事が生成・公開されたか）を記録し、重複投稿を防止する

自動化はあくまで「下書きまで」であり、**公開の最終判断は必ず人間が行う**（完全無人公開はスコープ外）。

## Capabilities

### New Capabilities

- `dev-activity-collection`: Git コミット・OpenSpec チェンジ・タスク進捗など開発中に蓄積した情報を期間指定で収集し、構造化ダイジェストとして出力する
- `blog-draft-generation`: 開発ダイジェストからブログ記事下書き（Markdown + frontmatter）を生成する。版管理されたテンプレート・スタイルガイド・品質ルーブリックに基づく多段生成、図解・埋め込みの自動付与、機械検証可能な記事リンターを含む
- `publication-content-safety`: 公開前コンテンツの機密情報・個人情報検査と、検出時のブロック・修正フロー
- `blog-publishing`: 人間承認を経た公開ワークフロー（PR ベースのレビュー、公開先アダプタ、公開履歴の記録、重複防止）

### Modified Capabilities

（なし — 既存インフラ系ケーパビリティの要件変更は伴わない）

## Impact

- 新規コード: 収集・生成・検査用の CLI スクリプト群（`tools/devblog/` 想定）と CI ワークフロー
- 外部依存: LLM API（下書き生成、コスト発生）、ブログ公開先（Git ベース連携を第一候補: Zenn GitHub 連携または静的サイト）
- シークレット: LLM API キーを CI シークレットとして管理（ハードコード禁止）
- 既存システムへの影響: アプリ本体（`app/`）・Supabase スキーマへの変更なし。cicd-pipeline の既存要件は変更せず、独立したワークフローとして追加する
