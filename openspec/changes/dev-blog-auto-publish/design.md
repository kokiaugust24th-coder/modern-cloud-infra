# Design: dev-blog-auto-publish

## Context

本リポジトリには開発情報が複数の形で蓄積されている:

- Git コミット履歴（コンベンショナルコミット形式、日本語メッセージ）
- OpenSpec チェンジ（`openspec/changes/`: proposal / design / specs / tasks、アーカイブ含む）
- 仕様本体（`openspec/specs/`: 9 つのインフラ系ケーパビリティ他）

これらは知識共有・技術発信の素材として価値が高いが、現状は発信につながっていない。手動執筆は継続コストが高いため、下書き生成までを自動化し、公開判断のみ人間が行うパイプラインを設計する。

## Goals / Non-Goals

**Goals:**

- 期間指定で開発情報を収集し、機械可読な「開発ダイジェスト」（JSON）として出力する
- ダイジェストから日本語のブログ記事下書き（Markdown + frontmatter）を LLM 支援で生成する
- 公開前に機密情報・個人情報の混入を機械的に検査し、検出時は公開をブロックする
- PR ベースの人間レビューを経て公開し、公開履歴を記録して重複投稿を防ぐ
- 定期実行（週次想定）と手動実行の両方をサポートする
- 人間の作業を「日々のメモ書き（任意）」と「PR レビュー」の 2 点のみに絞りつつ、公開水準の記事品質を既定で出せるようにする

**Non-Goals:**

- 完全無人での公開（人間承認の省略）は行わない
- ブログ基盤（サイトそのもの）の構築はこの変更のスコープ外。公開先はアダプタで抽象化し、既存の Git 連携型プラットフォームを利用する
- アプリ本体（`app/`）のユーザーデータ・セッションデータの発信は対象外（プロダクトのデータポリシー上、内省データは公開素材にしない）
- SEO 最適化、アクセス解析、多言語展開

## Decisions

### D1: 公開先は Git ベース連携（Zenn GitHub 連携を第一候補）

**決定**: 記事は公開用リポジトリ（またはディレクトリ）への Markdown コミットとして公開する。第一実装は Zenn の GitHub 連携。公開先はアダプタインターフェースで抽象化する。

**理由**: (a) PR レビュー = 人間承認ゲートを Git の既存機能で実現でき、追加の承認 UI が不要。(b) 公開用 API キーの管理が不要（GitHub 権限に集約）。(c) 公開履歴が Git 履歴として自然に残る。

**代替案**: WordPress / はてなブログ等の REST API 直接投稿 → 承認フローと履歴管理を自前実装する必要があり、API キー管理も増えるため見送り。アダプタ追加で将来対応可能。

### D2: パイプラインは「収集 → 生成 → 検査 → PR 作成」の 4 段直列、各段は独立 CLI

**決定**: `tools/devblog/` に collect / generate / scan / publish の 4 コマンドを置き、それぞれ単独実行可能にする。CI（GitHub Actions、週次 cron + 手動 dispatch）はこれらを直列に呼ぶだけとする。

**理由**: 各段が独立していればローカルで再現・デバッグ可能（再現可能性の方針に一致）。生成だけやり直す、検査だけ再実行するといった運用が容易。

**代替案**: 単一モノリシックスクリプト → 途中失敗時に全段やり直しになり、LLM コストも無駄になるため見送り。

### D3: 収集対象は Git 履歴と OpenSpec 成果物のみ（コード本文は含めない）

**決定**: ダイジェストの情報源はコミットメタデータ（ハッシュ・メッセージ・変更ファイル名・統計）と OpenSpec の proposal / design / tasks の本文に限定する。ソースコード本文・diff 本文は含めない。

**理由**: 機密混入の主経路（コード内のキー、内部 URL、`.env` 断片）を入口で遮断でき、後段のサニタイズゲートを多層防御の 2 層目にできる。記事の主題は「何を・なぜ作ったか」であり、コード断片が必要な場合は人間がレビュー時に追記する方が安全。

**代替案**: diff 全文を LLM に渡す → 記事の具体性は上がるが機密混入リスクとトークンコストが増大するため見送り。

### D4: サニタイズゲートは決定的ルール（正規表現 + 既知パターン）を必須層とし、LLM 判定は補助

**決定**: 公開ブロックの判定は決定的なパターン検査（API キー形式、メールアドレス、内部ホスト名、`.env` 由来値との照合等）が担う。LLM による「公開して問題ないか」の判定は警告（ラベル付与）に留め、単独でブロック解除の根拠にしない。

**理由**: 安全判定を非決定的なコンポーネントに依存させない（本プロダクトの「独立ゲート優先」の原則と同型）。誤検出は人間レビューで解除できるが、見逃しは回収できない。

**代替案**: gitleaks / trufflehog 等の既存 OSS をそのまま利用 → 採用する（車輪の再発明をしない）。既存ツール + プロジェクト固有パターンの追加定義という構成にする。

### D5: 公開履歴は公開用リポジトリ内の台帳ファイル（JSON Lines）で管理

**決定**: 「どの期間・どの情報源から、どの記事が生成・公開されたか」を `published.jsonl` に追記記録し、collect 段が参照して収集済み期間をスキップする。

**理由**: DB 追加は過剰。Git 管理された追記型台帳で監査可能性と重複防止の両方を満たせる。

### D6: 記事品質は「版管理された基準 + 機械検証」で担保し、LLM 自己批評は補助

**決定**: 記事テンプレート・スタイルガイド・品質ルーブリック（読者価値、構成、タイトル、具体性、日本語品質の観点）をリポジトリで版管理し、生成プロンプトに注入する。生成後は機械検証可能な記事リンター（frontmatter 必須項目、見出し階層、文字数範囲、リンク到達性、textlint による日本語校正）を必須ゲートとし、これに合格しない下書きは PR 化しない。LLM によるルーブリック自己採点は PR 本文への参考情報の添付に留める。

**理由**: 「高品質」を人の感覚ではなく検証可能な基準に落とすことで、品質が生成モデルの気まぐれに依存しない。サニタイズゲート（D4）と同じ「決定的検査を必須層、LLM 判定を補助」の構造で一貫する。

**代替案**: LLM 採点のみで品質判定 → 採点自体が非決定的で、基準の改善も追跡できないため見送り。

### D7: 多段生成（構成案 → 本文 → 自己批評リビジョン）

**決定**: 1 回の呼び出しで完成記事を出させるのではなく、(1) ダイジェストから構成案（アウトライン・想定読者・主題）を生成、(2) 構成案に沿って本文を生成、(3) ルーブリックに基づく自己批評と修正、の 3 段で生成する。各段の中間成果物は保存し、失敗時は該当段からやり直せるようにする。

**理由**: 段階分割は記事の構成品質を安定させる定石であり、中間成果物の保存は再現可能性の方針とも一致する。トークンコストは増えるが週次 1 記事では許容範囲（tasks で実測確認）。

**代替案**: 単発生成 → 構成の破綻や冗長さが出やすく、リビジョンの根拠も残らないため見送り。

### D8: 図解・埋め込みはテキスト表現に限定して自動化する

**決定**: 記事に自動で付与するビジュアル要素は、mermaid 図（アーキテクチャ・フロー等）と情報源リンクカード（コミット・PR・チェンジの URL）、および開発メモ由来の埋め込み記法（X ポスト・YouTube 等、公開先アダプタが変換）に限定する。画像ファイル（スクリーンショット等）の自動生成・自動選定は行わず、必要な場合は人間が PR レビュー時に追加する。

**理由**: テキスト表現ならサニタイズゲート（D4）を素通しできる。画像は内容の決定的検査ができず、機密（画面内の情報）混入リスクを機械的に排除できない。

### D9: 開発メモファイルを任意の素材源とする

**決定**: 開発者が任意で書き溜めるメモファイル（`devlog/` 配下、日付付き Markdown。学び・ハマりどころ・参照 URL・埋め込み候補）を収集対象に加える。メモは必須ではなく、存在しない週はコミット・OpenSpec 情報のみで記事を生成する。

**理由**: コミットメッセージだけでは「なぜ・何を学んだか」という記事の核になる情報が薄い。日々 1〜2 行のメモという最小の人手で記事品質を大きく上げられる。

### D10: LLM プロバイダは差し替え可能なアダプタとし、既定は GLM-5.2（コスパ重視）

**決定**: `generate` の LLM 呼び出しは `LlmClient` インターフェース（`complete({system, prompt, maxTokens}) -> {text, inputTokens, outputTokens}`）越しに行い、実装は `src/generate/providers/` に切り出す（`anthropic.ts`・`openaiCompatible.ts`・選択を行う `factory.ts`）。使用プロバイダ・モデル・APIキーの参照先はすべて `devblog.config.json` の `llm.provider`/`llm.model`/`llm.baseUrl`/`llm.apiKeyEnvVar` で指定し、コード変更なしに切り替えられるようにする。既定プロバイダは GLM-5.2（Z.ai、OpenAI 互換エンドポイント）とする。

**理由**: 週次1記事という低頻度・低ボリュームの用途では、生成に使う LLM 自体の性能差より運用コストの低さを優先すべきと判断した。2026-07-12 時点の公式価格調査では GLM-5.2（$1.4/$4.4 per 1M tok）が Claude Haiku 4.5（$1/$5）と同水準〜やや安く、Grok 4.5・GLM系フラッグシップ（$2/$6〜）は本用途には過剰スペックだった。将来さらに安い/高品質なモデルが登場した際に config だけで切り替えられるよう、LLM 呼び出しをプロバイダ非依存のインターフェースに切り出した。

**代替案**: 特定プロバイダの SDK に直接依存する実装 → 初期実装は速いが、モデル入れ替えのたびにコード変更・再テストが必要になるため見送り。`LlmClient` インターフェース越しにすることで既存の3段生成（D7）・記事リンター（D6）はプロバイダに関知しない。

## Risks / Trade-offs

- [LLM が事実と異なる内容（やっていない実装、誇張）を生成する] → ダイジェストに含まれる事実のみを記述するようプロンプトで制約し、記事末尾に情報源（コミット範囲・チェンジ名）を機械的に付記。最終的には人間レビューで担保
- [サニタイズの見逃しにより機密が公開される] → D3（入口制限）+ D4（決定的検査）+ PR レビューの 3 層防御。万一公開された場合は該当キーの即時ローテーションを運用手順に明記
- [週次実行が形骸化し、レビューされない PR が滞留する] → PR には自動で要約とチェックリストを付与し、一定期間レビューされない場合は自動クローズ（公開されないだけで損失なし）
- [LLM API コストの増加] → 週次 1 記事・入力はダイジェストのみのため小さい（試算を tasks で確認項目化）。実行毎のトークン使用量を台帳に記録
- [Zenn 側の仕様変更で連携が壊れる] → アダプタ境界を維持し、出力の本質は「frontmatter 付き Markdown」という汎用形式に留める
- [品質リンターが厳しすぎて下書きが常に落ち、記事が出ない] → リンター基準は版管理し、失敗理由を実行ログで可視化。基準の調整はコミットとして追跡する
- [開発メモが書かれず記事が痩せる] → メモは任意とし、無くてもコミット + OpenSpec 情報で成立する構成テンプレートを既定にする

## Open Questions（解消済み）

- **公開用リポジトリの形態**: 別リポジトリを採用する（例: `kochan-um/devblog`）。本リポジトリ（プロダクト本体）と公開用リポジトリの権限・シークレットを分離できるため。実際のリポジトリ作成・Zenn 連携設定は [docs/devblog-setup.md](../../../docs/devblog-setup.md) に手順化した人間の作業（アカウント操作を伴うため自動化しない）
- **著者名義**: `git config user.name`（本リポジトリでは `kochan-um`）をデフォルトの著者名義とし、記事テンプレートの frontmatter に固定値として設定する。変更したい場合はテンプレートを編集する
- **ライセンス表記**: 既定は「無断転載・無断利用を禁止し、引用は出典明記の上で可」とする一文を記事フッターに機械的に付記する（Zenn の一般的な慣行に合わせる）。将来 CC BY 等に変更したい場合はテンプレートの該当セクションを更新する
