# blog-draft-generation

## ADDED Requirements

### Requirement: 開発ダイジェストからの記事下書き生成
システムは、開発ダイジェスト（JSON）を入力として、日本語のブログ記事下書きを frontmatter 付き Markdown として生成しなければならない（MUST）。frontmatter にはタイトル・生成日時・対象期間・情報源（コミット範囲、OpenSpec チェンジ名）・下書きステータスを含めなければならない（MUST）。

#### Scenario: 下書きの生成
- **WHEN** 有効な開発ダイジェストを入力として生成コマンドを実行する
- **THEN** タイトル・導入・本文・まとめを備えた日本語 Markdown 記事が、必須 frontmatter フィールド付きで出力される

#### Scenario: 空ダイジェストの拒否
- **WHEN** 活動が空のダイジェストを入力として生成コマンドを実行する
- **THEN** 記事は生成されず、エラーではなく「生成対象なし」を示す結果が返る

### Requirement: 事実への接地
生成される記事は、入力ダイジェストに含まれる事実のみに基づかなければならない（MUST）。ダイジェストに存在しない実装・成果・数値を記事に含めてはならない（MUST NOT）。記事末尾には情報源一覧（コミット範囲・チェンジ名）を機械的に付記しなければならない（MUST）。

#### Scenario: 情報源の付記
- **WHEN** 記事下書きが生成される
- **THEN** 記事末尾に、生成に使用したコミット範囲と OpenSpec チェンジ名の一覧が LLM の出力とは独立した後処理で付記される

#### Scenario: 生成メタデータの記録
- **WHEN** 記事下書きが生成される
- **THEN** 使用モデル名・トークン使用量・生成日時が生成ログに記録される

### Requirement: 版管理された記事テンプレート・スタイルガイド・品質ルーブリック
システムは、記事テンプレート（構成の雛形）、スタイルガイド（トーン・表記・日本語規約）、品質ルーブリック（読者価値・構成・タイトル・具体性・日本語品質の観点）をリポジトリ内で版管理し、生成時に適用しなければならない（MUST）。生成された記事の frontmatter には使用したテンプレート・ルーブリックの版を記録しなければならない（MUST）。

#### Scenario: テンプレートとスタイルガイドの適用
- **WHEN** 記事下書きを生成する
- **THEN** 記事はテンプレートが定める構成（タイトル・導入・本文・まとめ・情報源）に従い、frontmatter に適用したテンプレート・ルーブリックの版が記録される

#### Scenario: 基準変更の追跡
- **WHEN** テンプレートまたはルーブリックを変更する
- **THEN** 変更はコミットとして記録され、以後の生成記事の frontmatter から使用版を遡って特定できる

### Requirement: 多段生成
記事の生成は、(1) 構成案（アウトライン・想定読者・主題）の生成、(2) 構成案に沿った本文の生成、(3) 品質ルーブリックに基づく自己批評と修正、の段階を経なければならない（MUST）。各段の中間成果物は保存され、失敗時には該当段からの再実行が可能でなければならない（MUST）。

#### Scenario: 三段の生成と中間成果物の保存
- **WHEN** 記事下書きの生成を実行する
- **THEN** 構成案・初稿・自己批評結果と修正稿がそれぞれ成果物として保存され、最終稿が下書きとして出力される

#### Scenario: 途中段からの再実行
- **WHEN** 本文生成段で失敗した後に再実行する
- **THEN** 保存済みの構成案を再利用して本文生成段から処理が再開される

### Requirement: 図解と情報源リンクの自動付与
生成される記事には、ダイジェストの内容に基づく mermaid 図（アーキテクチャ・フロー等、内容に図解が適する場合）と、情報源リンクカード（コミット・PR・OpenSpec チェンジの URL）を自動で含めなければならない（MUST）。画像ファイル（スクリーンショット等）の自動生成・自動選定は行ってはならない（MUST NOT）。

#### Scenario: mermaid 図の付与
- **WHEN** ダイジェストに構造的な変更（複数コンポーネントにまたがる実装等）が含まれる状態で記事を生成する
- **THEN** 記事には該当構造を表す mermaid コードブロックが含まれ、テキストとして機密検査の対象になる

#### Scenario: 画像ファイルの非生成
- **WHEN** 記事下書きが生成される
- **THEN** 下書きには画像ファイルへの新規参照は含まれず、画像が必要な場合は PR レビュー時に人間が追加する

### Requirement: 機械検証可能な記事品質ゲート
システムは、生成された下書きに対し決定的な記事リンターを実行しなければならない（MUST）。リンターは少なくとも frontmatter 必須項目、見出し階層の整合、本文文字数範囲、記事内リンクの到達性、日本語校正（textlint 等）を検証しなければならない（MUST）。リンターに合格しない下書きは公開フロー（PR 作成）に進めてはならない（MUST NOT）。LLM による品質採点は参考情報として添付してよいが、リンター不合格を上書きしてはならない（MUST NOT）。

#### Scenario: リンター合格による通過
- **WHEN** 生成された下書きがリンターの全検証項目に合格する
- **THEN** 検証結果が記録され、下書きは公開フローに進める状態になる

#### Scenario: リンター不合格によるブロック
- **WHEN** 下書きが検証項目（例: 本文文字数が範囲外、リンク切れ）に失敗する
- **THEN** 失敗項目と箇所がレポートされ、下書きは PR 作成に進まない

### Requirement: 生成失敗時の安全な停止
LLM API 呼び出しが失敗した場合、システムはタイムアウトと指数バックオフ付きリトライ（上限あり）を行い、上限到達時は不完全な下書きを出力せずに失敗として終了しなければならない（MUST）。

#### Scenario: API 障害時の挙動
- **WHEN** LLM API がリトライ上限まで失敗を返す
- **THEN** 部分的な記事ファイルは残らず、失敗理由がログに記録され、非ゼロ終了コードで終了する
