# blog-publishing

## ADDED Requirements

### Requirement: 人間承認を必須とする公開ワークフロー
システムは、検査に合格した記事下書きを公開用リポジトリへの Pull Request として提出しなければならない（MUST）。記事の公開は人間による PR の承認・マージによってのみ行われなければならず（MUST）、システムが自動でマージ・公開を完了してはならない（MUST NOT）。

#### Scenario: PR の自動作成
- **WHEN** 検査合格済みの記事下書きに対して公開コマンドを実行する
- **THEN** 公開用リポジトリに記事ファイルを含むブランチと PR が作成され、PR 本文には記事要約・対象期間・情報源・検査結果・レビューチェックリストが含まれる

#### Scenario: 自動マージの禁止
- **WHEN** 作成された PR がレビューされないまま存在する
- **THEN** システムはいかなる条件でも自動マージを行わず、一定期間（既定 30 日）レビューされない PR は公開されないままクローズされる

### Requirement: 公開先アダプタ
公開処理は公開先プラットフォームをアダプタインターフェースで抽象化しなければならない（MUST）。第一実装として Git ベース連携（frontmatter 付き Markdown のコミット）を提供しなければならない（MUST）。アダプタ固有の変換（frontmatter 形式、ディレクトリ規約）はアダプタ内に閉じなければならない（MUST）。

#### Scenario: Git ベース公開先への出力
- **WHEN** Git ベースアダプタを指定して公開コマンドを実行する
- **THEN** 公開先プラットフォームの規約（ディレクトリ構成・frontmatter スキーマ）に適合した Markdown ファイルが生成される

#### Scenario: 埋め込み記法の変換
- **WHEN** 記事に汎用形式の埋め込み参照（X ポスト URL・YouTube URL・リンクカード対象 URL）が含まれる状態で Zenn アダプタが変換を行う
- **THEN** それぞれ Zenn の埋め込み記法（`@[tweet]`、`@[youtube]`、リンクカード）に変換されて出力される

### Requirement: 公開履歴の記録と重複防止
システムは、公開（PR マージ）された記事について、記事識別子・公開日時・対象期間・情報源を追記型の公開履歴台帳に記録しなければならない（MUST）。同一の期間・情報源の組に対する重複した PR 作成を既定で拒否しなければならない（MUST）。

#### Scenario: 公開の記録
- **WHEN** 記事 PR がマージされる
- **THEN** 公開履歴台帳に記事識別子・公開日時・対象期間・情報源コミット範囲が追記される

#### Scenario: 重複 PR の拒否
- **WHEN** 既に PR 作成済み（未クローズ）または公開済みの期間・情報源に対して公開コマンドを実行する
- **THEN** 新しい PR は作成されず、既存の PR または公開済み記事への参照が提示される

### Requirement: 定期実行と手動実行
パイプライン全体（収集 → 生成 → 検査 → PR 作成）は、スケジュール実行（既定: 週次）と手動実行の両方で起動できなければならない（MUST）。いずれの起動方法でも同一の検査・承認要件が適用されなければならない（MUST）。

#### Scenario: 週次スケジュール実行
- **WHEN** 週次スケジュールでパイプラインが起動し、対象期間に開発活動が存在する
- **THEN** 収集から PR 作成までが自動実行され、結果（作成された PR、またはスキップ理由）が実行ログに記録される

#### Scenario: 手動実行時の同一ゲート適用
- **WHEN** 開発者が期間を指定して手動でパイプラインを起動する
- **THEN** スケジュール実行と同一の機密検査・人間承認要件が適用される
