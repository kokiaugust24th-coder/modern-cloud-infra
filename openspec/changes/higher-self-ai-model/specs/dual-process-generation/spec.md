# dual-process-generation: 二重過程応答生成

## ADDED Requirements

### Requirement: エゴプロセスによる候補生成
システムは、ユーザー入力と信念状態メモリのコンテキストから、k個(既定k=4)の応答候補を生成しなければならない(SHALL)。候補は多様性を確保するため、意味的類似度が閾値を超える候補ペアを含んではならない(MUST NOT)。

#### Scenario: 候補の多様性確保
- **WHEN** 生成されたk候補のうち2つの意味的類似度が閾値を超える
- **THEN** 片方を破棄して再サンプリングし、最終的にk個の相互に異なる候補を揃える

### Requirement: ハイヤーセルフプロセスによる評価と選択
システムは、各候補を(1)感情ヴェイランス遷移の適切さ、(2)信念状態との整合、(3)内発的動機スコアの3軸で評価し、統合スコア最大の候補を選択しなければならない(SHALL)。全候補が最低基準を下回る場合は、最良候補に書き換え指示を付けてエゴプロセスに差し戻さなければならない(MUST)。

#### Scenario: 通常の選択
- **WHEN** k候補のうち少なくとも1つが3軸すべての最低基準を満たす
- **THEN** 統合スコア最大の候補が最終応答として選択される

#### Scenario: 差し戻しループ
- **WHEN** 全候補がいずれかの軸で最低基準を下回る
- **THEN** 評価器は不足軸を明示した書き換え指示を生成し、エゴプロセスが再生成する(最大2回、超過時は安全な定型応答にフォールバック)

### Requirement: 選択理由の説明可能性
システムは、最終応答について「どの軸のどのスコアにより選択されたか」を構造化ログとして記録しなければならず(MUST)、ユーザーが求めた場合は自然言語で選択理由を提示できなければならない(SHALL)。

#### Scenario: 内部監査ログ
- **WHEN** 応答が確定する
- **THEN** 全候補の3軸スコア、選択された候補ID、差し戻し履歴が監査ログに記録される

#### Scenario: ユーザーへの理由開示
- **WHEN** ユーザーが「なぜその提案なの?」と選択理由を尋ねる
- **THEN** システムは感情状態・信念・動機の観点から選択理由を平易な言葉で説明する

### Requirement: レイテンシ上限
二重過程を経た応答生成のp95レイテンシは、単一プロセス生成の3倍以内でなければならない(MUST)。

#### Scenario: レイテンシ退行の検出
- **WHEN** 継続的負荷試験でp95レイテンシが基準の3倍を超える
- **THEN** デプロイはブロックされ、候補数kまたは評価器の軽量化が検討される
