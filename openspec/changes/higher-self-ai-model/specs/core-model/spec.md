# core-model: 自社基盤モデル

## ADDED Requirements

### Requirement: ベースモデルのライセンス適格性
コアモデルの基盤となる公開重みモデルは、商用利用・改変・再配布を許諾するライセンス(Apache-2.0等)を持つものでなければならない(MUST)。ライセンス調査結果はADRとして記録しなければならない(MUST)。

#### Scenario: ライセンス不適格なモデルの排除
- **WHEN** 候補ベースモデルのライセンスが商用利用または重み改変を制限している
- **THEN** そのモデルは選定候補から除外され、除外理由がADRに記録される

### Requirement: 領域コーパスによる継続事前学習
システムは、心理学(感情モデル、認知行動理論)、コーチング対話、内省的ライティングを含む領域コーパスでベースモデルを継続事前学習しなければならない(SHALL)。コーパスは出所・ライセンス・品質スコアのメタデータを持ち、重複除去と個人情報除去を経なければならない(MUST)。

#### Scenario: データ品質ゲート
- **WHEN** 収集した文書が品質スコア閾値未満、または個人情報検出器にヒットする
- **THEN** その文書は学習コーパスから除外され、除外統計がデータカードに記録される

#### Scenario: 継続事前学習の退行検知
- **WHEN** 継続事前学習後のモデルが汎用ベンチマークでベースモデル比5%超の性能低下を示す
- **THEN** 学習レシピ(データ混合比・学習率)を見直し、退行が解消するまでリリースしない

### Requirement: 選好最適化による応答様式の学習
システムは、SFTの後にDPO系の選好最適化を行い、「傾聴→反映→問いかけ」の対話様式と、断定的な人生指示の回避を学習させなければならない(SHALL)。選好データは自社作成し、アノテーション指針をバージョン管理しなければならない(MUST)。

#### Scenario: 断定的指示の抑制
- **WHEN** ユーザーが「仕事を辞めるべきですか」のような人生の重大決定を委ねる質問をする
- **THEN** モデルは決定を代行せず、ユーザー自身の判断材料を整理する応答を生成する

### Requirement: 推論APIの内部状態公開
推論サービングは、応答トークンに加えて、二重過程アーキテクチャが必要とする内部情報(候補群、対数尤度、隠れ状態への任意アクセス)を内部APIとして公開しなければならない(MUST)。

#### Scenario: 候補生成モードでの呼び出し
- **WHEN** 内部クライアントが温度・候補数kを指定して候補生成モードで呼び出す
- **THEN** サービングはk個の応答候補と各候補のメタデータ(対数尤度等)を返す

### Requirement: 評価ハーネスの常設
システムは、感情誘導品質・安全性・汎用性能を測る自動評価ハーネスを持ち、すべてのモデル更新はハーネス通過をリリース条件としなければならない(MUST)。

#### Scenario: リリースゲート
- **WHEN** 新しいモデルチェックポイントが評価ハーネスのいずれかの必須項目で基準を下回る
- **THEN** そのチェックポイントは本番デプロイ対象から除外され、失敗項目がレポートされる
