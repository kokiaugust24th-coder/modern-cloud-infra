# Design: modern-cloud-infra

## Context

現状、このプロジェクトにはクラウドインフラに関する明文化された仕様が存在しない。本変更では、モダンなクラウドインフラのアーキテクチャ原則と要件を高レベル仕様として定義する。仕様はプロバイダ非依存の要件として記述し、参照実装として AWS を想定するが、GCP / Azure へも同じ要件をマッピングできる粒度に保つ。

## Goals / Non-Goals

**Goals:**
- インフラ構築・運用の判断基準となる高レベル要件を 7 つのケーパビリティとして定義する
- 再現性（IaC）、回復性（マルチAZ・DR）、安全性（多層防御・最小権限)、可観測性を仕様レベルで担保する
- 各要件をテスト可能なシナリオ（WHEN/THEN）として表現する

**Non-Goals:**
- 特定リソースの詳細設計（インスタンスサイズ、CIDR の具体値、コスト見積り）
- Terraform モジュールや Kubernetes マニフェストの実装（後続の変更で実施）
- マルチリージョン・アクティブ/アクティブ構成（DR はバックアップ＋リストア／ウォームスタンバイまでを対象）
- オンプレミスとのハイブリッド接続

## Decisions

### D1: プロバイダ非依存の要件 + AWS 参照実装（確定）
仕様は「マネージドコンテナオーケストレーション」のような抽象度で記述し、参照実装のプロバイダは **AWS に確定**する（2026-07-11 決定）。
- 理由: 仕様の寿命はツールより長い。プロバイダ移行やマルチクラウド判断時に仕様を書き直さずに済む。実装は AWS で行うが、要件は他プロバイダにもマッピング可能な抽象度を維持する。
- 代替案: AWS 固有仕様 — 具体性は上がるがロックインし、仕様が実装詳細に汚染される。却下。

### D2: IaC は宣言的ツール（Terraform/OpenTofu 系)を第一選択
状態ファイルをリモート管理し、すべての変更をプルリクエスト経由にする。手動コンソール変更は禁止（ドリフト検出で検知）。
- 理由: 宣言的 + 状態管理により再現性と差分レビューが成立する。
- 代替案: CloudFormation/CDK — プロバイダ固定になるため D1 と整合しない。

### D3: コンピュートはコンテナ第一、イベント駆動はサーバーレス
常駐ワークロードはマネージド Kubernetes（例: EKS）、イベント駆動・低頻度処理は FaaS（例: Lambda）を使う二本立て。
- 理由: 全部 Kubernetes は運用過剰、全部サーバーレスは長時間・常駐処理に不向き。ワークロード特性で使い分けるのが現代の標準。
- 代替案: ECS/Fargate 単独 — シンプルだがエコシステム（GitOps、オートスケーラ）の選択肢が狭い。要件上は「マネージドオーケストレーション」なので実装時に再検討可。

### D4: デプロイは GitOps（プル型）
Git リポジトリを望ましい状態の唯一の情報源とし、クラスタ側エージェント（例: Argo CD）が同期する。
- 理由: 監査可能性（全変更が Git 履歴）、ロールバックが revert で完結、CI からクラスタへの強い認証情報を排除できる。
- 代替案: CI からのプッシュ型デプロイ — シンプルだが認証情報の集中とドリフトの検出不能が弱点。

### D5: 可観測性は OpenTelemetry 準拠で収集を統一
ログ・メトリクス・トレースの計装は OpenTelemetry 標準に準拠し、バックエンド（マネージド監視サービス等）は差し替え可能にする。
- 理由: 計装のベンダーロックインを避け、バックエンド選定を独立した判断にできる。

### D6: セキュリティはゼロトラスト志向のベースライン
人・ワークロードとも最小権限の IAM、シークレットは専用ストアで管理し環境変数への平文埋め込みを禁止、保存時・転送時とも暗号化を必須とする。
- 理由: 侵害前提（assume breach）の設計が現代の標準。後付けでの導入はコストが跳ね上がるため仕様段階で固定する。

### D7: 参照技術スタック（確定）
2026-07-11 に以下のスタックを確定した。各ケーパビリティの実装はこのスタックを既定とする。

| 領域 | 選定 | 補足 |
|---|---|---|
| クラウド | AWS | 参照実装プロバイダ（バックエンド・データ層） |
| CDN / DNS / WAF / フロント配信 | Cloudflare | 全フェーズ共通のエッジ層。フェーズ1 以降は AWS オリジン（ALB 等）の前段に置く |
| IaC | Terraform / OpenTofu | 状態は S3 バックエンド + ロック、環境分離はディレクトリ/ワークスペース分割 |
| コンテナ基盤 | EKS（マネージド Kubernetes） | ノードは Karpenter によるオートスケールを想定 |
| サーバーレス | Lambda | イベント駆動・低頻度処理用（D3 のとおり） |
| コンテナレジストリ | ECR | イメージスキャン有効化 |
| GitOps | Argo CD | Flux との比較で UI・多環境管理・普及度から選定。実装開始前に再確認可 |
| CI | GitHub Actions | ビルド・テスト・スキャン。デプロイは Argo CD に委譲（プッシュ型にしない） |
| 可観測性 | CloudWatch + AWS Distro for OpenTelemetry (ADOT) | 計装は OTel 準拠（D5）。将来 Grafana スタック等へ移行可能 |
| データベース | RDS / Aurora（マルチAZ） | PITR 有効化 |
| オブジェクトストレージ | S3 | ライフサイクルポリシー必須 |
| シークレット | AWS Secrets Manager | ワークロードからは IRSA（IAM Roles for Service Accounts）で取得 |
| 監査 | CloudTrail（全リージョン・改ざん防止保管） | security-baseline の監査要件に対応 |

- 理由: 「まず動く標準構成を最小の運用負荷で」を優先。監視はクラウド標準（CloudWatch）から始め、OTel 準拠計装によりバックエンド移行の自由度を残す。
- 代替案: 監視に Grafana スタック／Datadog — 高機能だが初期の運用負荷またはコストが大きい。規模拡大時に再評価する。

### D8: 無料から有料への段階的スケール戦略（確定）
インフラは 3 フェーズで導入する。前フェーズの成果物（アプリコード・スキーマ・IaC）を書き直さずに次フェーズへ移行できることを `workload-portability` スペックで担保する。

| フェーズ | 想定時期 | 構成 | 月額目安 |
|---|---|---|---|
| **0: 無料検証期** | MVP〜初期ユーザー | Cloudflare Pages（フロント）+ Supabase（Postgres/認証）の無料プラン | ¥0 |
| **1: クレジット期** | ユーザー増加・調達後 | バックエンドのみ AWS へ移行: ECS Fargate（または Lambda コンテナ）+ Aurora Serverless v2 + S3。フロントは Cloudflare 残置。AWS Activate 等のクレジットで実質無料運用 | クレジットで相殺 |
| **2: 本番フル構成** | 事業本格化 | D7 の確定スタック（EKS + Argo CD + RDS/Aurora マルチAZ）。Cloudflare は DNS/CDN/WAF として前段に残置 | 従量 + Savings Plans |

フェーズ0 に Cloudflare Pages + Supabase を選定した理由:
- Cloudflare Pages の無料プランは静的配信の帯域・リクエスト数が無制限で、無料枠の天井が最も高い
- Cloudflare はフェーズ1 以降も DNS / CDN / WAF として AWS の前段に残すため、**フロントの移行工程そのものが発生しない**（移行対象はバックエンドとデータのみになる）
- Supabase の実体は標準 PostgreSQL であり、`pg_dump` で Aurora/RDS へそのまま移行できる（データモデルの書き直しが発生する Firebase/DynamoDB 系を回避）
- フロントは静的第一（Vite SPA / Astro）とする。SSR が必要な場合はアダプター方式のフレームワーク（Astro / Remix）で Pages Functions を利用し、ロジック本体はランタイム非依存に保つ
- 制約: Workers 固有ストレージ（KV / D1 / Durable Objects）をプライマリデータストアにしてはならない（workload-portability の Postgres 互換要件で拘束済み）
- 代替案: Vercel — Next.js 統合は最良だが付加機能の粘着性が高く、エッジ層として残す用途もない。AWS 無料枠のみの構成 — DB 無料化に DynamoDB 依存が必要で可搬性を損なう。いずれも却下。

**フェーズ移行のトリガー**（cost-management スペックの監視要件と対応。数値は 2026-07-11 に実アカウントで確認した無料枠上限）:

| 無料枠 | 上限 | 70% 到達の目安 |
|---|---|---|
| Supabase DB 容量 | 500MB | 350MB |
| Supabase MAU（認証） | 50,000 | 35,000 |
| Cloudflare Pages 帯域・リクエスト | 無制限 | 対象外（天井なし） |
| Cloudflare R2 ストレージ | 10GB/月 | 7GB |
| Cloudflare R2 Class A operations | 100万回/月 | 70万回 |
| Cloudflare R2 Class B operations | 1,000万回/月 | 700万回 |

- 0→1: 上記いずれかの無料枠使用率が 70% に到達、または Supabase/Cloudflare の有料プラン費用が AWS 移行後見積りを超える見込みのとき
- 1→2: クレジット残高の消化予測が 90 日を切ったとき、または単一サービスのスケール要件（常駐ワークロード数・チーム数）が軽量コンテナ基盤の運用限界を超えたとき

**フェーズ別のスペック適用開始**:

| スペック | フェーズ0 | フェーズ1 | フェーズ2 |
|---|---|---|---|
| workload-portability | ● 全面適用 | ● | ● |
| cost-management | ● 全面適用 | ● | ● |
| infrastructure-as-code | ○ 設定・スキーマのコード管理のみ | ● 全面適用 | ● |
| security-baseline | ○ シークレット管理・最小権限のみ | ● 全面適用 | ● |
| observability | ○ 構造化ログ・OTel 計装のみ | ● 全面適用 | ● |
| cicd-pipeline | ○ 品質ゲートのみ | ○ + 環境昇格 | ● GitOps 含め全面適用 |
| data-persistence | ○ バックアップ検証のみ | ● 全面適用 | ● |
| network-foundation | −（PaaS 管轄） | ○ 最小構成 | ● 全面適用 |
| compute-platform | −（PaaS 管轄） | ○ 軽量コンテナ | ● K8s 要件含め全面適用 |

**フェーズ0 リファレンス構成**（2026-07-11 レビュー確定）:

| 部品 | 選定 | 備考 |
|---|---|---|
| フロント | Cloudflare Pages（Vite SPA / Astro、静的第一） | 帯域・リクエスト無制限。SSR は Pages Functions + アダプター方式 |
| API / ロジック | Supabase 自動 API + 薄い Pages Functions | ロジック本体はランタイム非依存の関数に置く（workload-portability） |
| DB / 認証 | Supabase（Postgres + Auth） | RLS を全テーブルで有効化（Aurora へそのまま移行可能） |
| CI | GitHub Actions | lint・型・テスト・シークレットスキャン・**コンテナビルド検証**をマージ条件に |
| IaC | Terraform（Cloudflare / Supabase プロバイダ）+ スキーママイグレーション | ダッシュボード手作業ゼロをフェーズ0 から実現 |
| 監視 | OTel 計装 + Grafana Cloud 無料プラン | D5 の OTel 準拠によりフェーズ1 で CloudWatch へバックエンドのみ差し替え |
| バックアップ | GitHub Actions 週次 cron で `pg_dump` → **Cloudflare R2**（無料枠 10GB、S3 互換）+ リストア検証 | AWS を導入しないフェーズ0 の原則に合わせ保管先も Cloudflare に統一。**フェーズ1 移行（pg_restore）の予行演習を兼ねる** |
| リモート状態管理 | Terraform Cloud 無料枠 | infrastructure-as-code の「ロック付きリモート状態」要件をフェーズ0 から満たす |

**フェーズ1 リファレンス構成と移行設計**:

- **変わらないもの**: フロント（Pages）、認証（Supabase Auth 残留）、CI、Terraform、OTel 計装、コンテナイメージ、DB スキーマ・RLS
- **移るもの**: アプリ実行 → ECS Fargate（ALB 経由）、DB → Aurora Serverless v2、ファイル → S3、シークレット → Secrets Manager、監視バックエンド → CloudWatch + ADOT
- **ネットワーク**: 最小 VPC（2AZ、パブリック/プライベート）。S3・ECR・Secrets Manager へは VPC エンドポイント経由とし、NAT ゲートウェイは作らないか 1 個に限定する（月額約 $32/個 の固定費回避）
- **認証の扱い**: A 案 = Supabase Auth 残留（無料 5万 MAU、認証抽象化層があるため独立して移行可能）を基本とし、無料枠接近または MAU 増加時に B 案 = Cognito 移行（AWS クレジット適用可）を評価する。判断は cost-management のクレジット対象外費用監視で駆動する
- **クレジット対象外費用**: Supabase Pro（$25/月〜）・Cloudflare 有料プランは AWS クレジットで相殺されない。実費として予算計上する（cost-management 参照）
- **フェーズ2 予行**: フェーズ1 期間中に CI で kind による Kubernetes マニフェスト検証を開始し、フェーズ2 の EKS 移行を「検証済みマニフェストの差し替え」にする

**カットオーバー手順（0→1）** — Cloudflare を残したことで切り替えとロールバックが即時になる:

1. Terraform で AWS 側を構築し、週次リハーサル済みの `pg_restore` でデータ投入して並行稼働
2. 短い書き込み停止時間に最終 `pg_dump` / `pg_restore` を実行。書き込み停止を許容できない場合は **Postgres 論理レプリケーション**（無料・標準機能）で Supabase → Aurora を継続同期し、切替時は数秒の書き込み凍結のみとする（代替: AWS DMS 無料枠 = Aurora への移行は 6 ヶ月無料、行単位の自動検証機能つき）
3. **整合性検証**: 移行前後でテーブルごとの行数と集約チェックサム（`md5(string_agg(...))` 等）を照合する。論理レプリケーション利用時はシーケンス値が複製されないため `setval` で手動同期し、移行期間中の DDL 変更は凍結する
4. Cloudflare のオリジンを Supabase 向けから ALB 向けへ切り替え（失敗時はオリジンを戻すだけで即ロールバック）
5. SLO 指標をダッシュボードで確認し、安定後に旧構成を縮退（Supabase は認証専用として残す）

**フェーズ2 リファレンス構成と移行設計**（2026-07-11 レビュー確定）:

- **変わらないもの**: Cloudflare エッジ、CI、Terraform、OTel 計装、コンテナイメージ、**Aurora のデータ（移行なし）**、S3、Secrets Manager、認証（A/B 案どちらでも独立）
- **変わるもの**:
  - 実行基盤: ECS Fargate → **EKS**（Karpenter によるノードオートスケール、マルチAZ 分散、IRSA）
  - デプロイ方式: CI からのプッシュ → **Argo CD による GitOps**（cicd-pipeline の全面適用）
  - DB: Aurora Serverless v2 → プロビジョンド or 高 ACU 上限 + **マルチAZ フェイルオーバー**（インプレース変更であり**データ移行は発生しない**）
  - ネットワーク: network-foundation 全面適用（3 層サブネット、フローログ、VPC エンドポイント + 必要最小限の NAT）
  - 可観測性: SLO / エラーバジェットベースのアラートまで全面適用
- **コスト最適化を移行と同時に導入**: Karpenter + Spot インスタンス、Graviton（ARM）ノード、非本番の夜間自動停止。安定稼働後に Savings Plans を契約
- **1→2 の最大の特徴**: データ移行が存在しないこと。動かすのはコンピュートとデプロイ方式のみで、マニフェストはフェーズ1 の CI（kind 検証）で熟成済みのものを使う

**カットオーバー手順（1→2）**:

1. Terraform で EKS + Argo CD を構築し、kind 検証済みマニフェストを Argo CD で同期
2. **同じ Aurora / S3 / Secrets Manager を参照する形で** EKS 側サービスを並行稼働させる（データの二重管理は発生しない）
3. ALB の加重ルーティング（または Cloudflare オリジン切り替え）で 5% → 50% → 100% と段階的にトラフィックを移行（カナリア）。エラー率悪化時は即座に比率を戻す
4. 安定後に ECS 側を縮退し、Savings Plans の契約検討を開始

## Risks / Trade-offs

- [抽象的な仕様が実装時に解釈ブレを生む] → 各要件にテスト可能なシナリオを付け、実装変更時は delta spec で具体化する
- [Kubernetes の運用負荷が小規模チームに過大] → 要件は「マネージドオーケストレーション」に留め、実装時に Fargate 等の低運用オプションを選べる余地を残す
- [7 ケーパビリティ全部の同時実装は大きすぎる] → 仕様定義と実装を分離。実装は network → security → compute の依存順で別変更に分割する
- [CloudWatch はクエリ性・ダッシュボード柔軟性に限界がある] → 計装を OTel 準拠にしてあるため、規模拡大時に Grafana スタック等へバックエンドのみ差し替え可能
- [フェーズ0 で PaaS 固有機能に依存し移行不能になる] → workload-portability スペック（データアクセス層分離・認証抽象化・コンテナ化可能ビルド・Postgres 互換）を フェーズ0 から全面適用する
- [クレジット切れと同時に高額請求が始まる] → cost-management スペックでクレジット消化の監視と 90 日前アラートを必須化し、移行トリガーに組み込む
- [フェーズ移行そのものが大工事になる] → 各フェーズの移行はデータ（pg_dump/restore）・コンテナイメージ・IaC モジュール差し替えの 3 点に限定されるよう可搬性要件で拘束する

## Migration Plan

本変更は仕様定義のみでランタイムへの影響はない。承認後 `openspec/specs/` に同期し、以降のインフラ実装変更はこの仕様を親とする。ロールバックは change のアーカイブ取り消しで完結する。

## Open Questions

- ~~参照実装のプロバイダ確定~~ → AWS に確定（D1/D7、2026-07-11）
- ~~GitOps ツール・監視バックエンドの選定~~ → Argo CD / CloudWatch+ADOT に確定（D7、2026-07-11）
