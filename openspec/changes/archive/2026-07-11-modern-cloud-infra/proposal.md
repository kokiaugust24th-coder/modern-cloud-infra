# Proposal: modern-cloud-infra

## Why

クラウドインフラの構成が暗黙知のまま場当たり的に構築されると、環境間の差異・セキュリティホール・復旧不能な障害を招く。モダンなクラウドインフラの高レベル仕様（アーキテクチャ原則と要件）を明文化し、プロビジョニングや運用の判断基準となる単一の信頼できる情報源を確立する。

## What Changes

- クラウドインフラ全体の高レベル仕様を新規に定義する（特定プロバイダに依存しない要件とし、参照実装は AWS を想定）
- すべてのリソースを Infrastructure as Code で宣言的に管理する方針を確立する
- マルチAZ・多層防御を前提としたネットワーク基盤の要件を定義する
- コンテナ／サーバーレスを第一級とするコンピュート基盤の要件を定義する
- GitOps ベースの CI/CD パイプラインとデプロイ戦略（カナリア／ブルーグリーン）の要件を定義する
- ログ・メトリクス・トレースの三本柱による可観測性の要件を定義する
- 最小権限・暗号化・シークレット管理・監査を含むセキュリティベースラインを定義する
- マネージドデータストアとバックアップ／DR の要件を定義する
- 無料構成（フェーズ0）から有料フル構成（フェーズ2）へ段階的に移行するスケール戦略を定義する
- 予算アラート・タグ付け・クレジット監視を含むコスト管理の要件を定義する
- フェーズ間の移行を書き直しなしで行うためのワークロード可搬性の要件を定義する

## Capabilities

### New Capabilities

- `infrastructure-as-code`: 全リソースの宣言的定義、状態管理、環境（dev/staging/prod）の分離と再現性
- `network-foundation`: マルチAZ の VPC 設計、パブリック／プライベートサブネット分離、通信制御
- `compute-platform`: コンテナオーケストレーションおよびサーバーレス実行基盤、オートスケーリング
- `cicd-pipeline`: GitOps によるビルド・テスト・デプロイの自動化、段階的リリースとロールバック
- `observability`: 構造化ログ・メトリクス・分散トレースの収集、SLO ベースのアラート
- `security-baseline`: IAM 最小権限、保存時・転送時の暗号化、シークレット管理、監査ログ
- `data-persistence`: マネージドデータベース／オブジェクトストレージ、バックアップと災害復旧
- `cost-management`: 予算アラート、コスト配分タグ、無料枠・クレジットの消化監視、フェーズ移行の判断基準
- `workload-portability`: データアクセス層の分離、認証の抽象化、コンテナ化可能なビルド、Postgres 互換データストア（フェーズ移行を書き直しなしで可能にする）

### Modified Capabilities

（なし — 既存スペックは存在しない）

## Impact

- 新規リポジトリ構成: `openspec/specs/` 配下に 9 つのケーパビリティスペックが追加される
- インフラの導入はフェーズ0（無料 PaaS）→ フェーズ1（AWS + クレジット）→ フェーズ2（EKS フル構成）の段階制となり、各スペックの適用開始フェーズは design.md で定義される
- 今後のインフラ実装（Terraform/OpenTofu モジュール、Kubernetes マニフェスト、CI/CD 定義）はこの仕様に準拠する
- 既存コードへの影響: なし（本変更は仕様定義のみで、実装は後続の変更で行う）
- 依存: クラウドプロバイダのアカウント、IaC ツールチェーン、コンテナレジストリ（実装フェーズで必要）
