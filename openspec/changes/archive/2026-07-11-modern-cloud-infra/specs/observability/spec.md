# observability

## ADDED Requirements

### Requirement: 構造化ログの集中管理
すべてのワークロードは構造化ログ（JSON 等）を標準出力へ出力し、ログは集中基盤へ収集・保管されなければならない (MUST)。ログには相関 ID（トレース ID）が含まれなければならない (MUST)。

#### Scenario: 横断的なログ検索
- **WHEN** 特定リクエストの障害を調査するとき
- **THEN** トレース ID で複数サービスにまたがるログを時系列で検索・追跡できる

### Requirement: メトリクスの収集
基盤（CPU・メモリ・ネットワーク）とアプリケーション（レイテンシ・スループット・エラー率）のメトリクスが収集され、ダッシュボードで可視化されなければならない (MUST)。計装は OpenTelemetry 標準に準拠しなければならない (SHALL)。

#### Scenario: サービス健全性の確認
- **WHEN** 運用者がサービスの状態を確認するとき
- **THEN** レイテンシ・トラフィック・エラー・飽和度（ゴールデンシグナル）がダッシュボードで参照できる

### Requirement: 分散トレーシング
サービス間のリクエストはトレースコンテキストを伝播し、分散トレースとして収集されなければならない (MUST)。

#### Scenario: ボトルネックの特定
- **WHEN** リクエストのレイテンシが悪化したとき
- **THEN** トレースからどのサービス・どの依存呼び出しに時間を要したかを特定できる

### Requirement: SLO ベースのアラート
主要サービスには SLO（可用性・レイテンシ）が定義され、アラートはエラーバジェットの消費に基づいて発報されなければならない (MUST)。アラートは対応手順（Runbook）への参照を含まなければならない (SHALL)。

#### Scenario: エラーバジェット消費時の発報
- **WHEN** エラーバジェットの消費速度が閾値を超過したとき
- **THEN** オンコール担当へ通知され、通知には影響範囲と Runbook への参照が含まれる
