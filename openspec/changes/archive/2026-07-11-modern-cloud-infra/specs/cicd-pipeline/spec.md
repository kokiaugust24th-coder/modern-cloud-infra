# cicd-pipeline

## ADDED Requirements

### Requirement: 自動ビルド・テストパイプライン
すべてのコード変更は、マージ前に自動化されたビルド・テスト・静的解析・脆弱性スキャンを通過しなければならない (MUST)。

#### Scenario: 品質ゲートによるブロック
- **WHEN** テスト失敗または重大な脆弱性を含む変更がプルリクエストとして提出されたとき
- **THEN** パイプラインが失敗し、マージがブロックされる

### Requirement: GitOps によるデプロイ
デプロイは Git リポジトリに記録された望ましい状態を情報源とし、環境側のエージェントが同期するプル型で行われなければならない (MUST)。CI システムは本番環境への直接的な強い認証情報を保持してはならない (MUST NOT)。

#### Scenario: Git を起点とした同期
- **WHEN** デプロイ設定リポジトリの本番向け定義が更新されたとき
- **THEN** 環境側エージェントが差分を検知し、実環境を定義された状態へ収束させる

#### Scenario: ロールバック
- **WHEN** デプロイ後に障害が検知され、直前のコミットへ revert されたとき
- **THEN** 環境は自動的に前バージョンの状態へ収束する

### Requirement: 段階的リリース
本番デプロイは、カナリアリリースまたはブルーグリーンにより段階的にトラフィックを移行し、健全性指標の悪化時には自動的に中断・巻き戻しされなければならない (MUST)。

#### Scenario: カナリアの自動中断
- **WHEN** カナリア版のエラー率が基準値を超過したとき
- **THEN** トラフィック移行は自動的に停止され、全トラフィックが安定版へ戻される

### Requirement: 環境昇格フロー
変更は dev → staging → prod の順に昇格されなければならない (MUST)。本番への昇格には承認記録が残らなければならない (MUST)。

#### Scenario: 本番昇格の承認
- **WHEN** staging で検証済みの変更を本番へ昇格するとき
- **THEN** 承認者・日時・対象バージョンが監査可能な形で記録された上でデプロイされる
