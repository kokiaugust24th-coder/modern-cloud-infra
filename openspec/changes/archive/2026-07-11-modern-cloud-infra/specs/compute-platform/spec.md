# compute-platform

## ADDED Requirements

### Requirement: コンテナ第一の実行基盤
常駐ワークロードは、マネージドなコンテナオーケストレーション基盤（例: マネージド Kubernetes）上でコンテナとして実行されなければならない (MUST)。コンテナイメージはバージョン付きでレジストリに保管され、イミュータブルに扱われなければならない (MUST)。

#### Scenario: コンテナによるデプロイ
- **WHEN** アプリケーションの新バージョンをデプロイするとき
- **THEN** タグ付けされたイメージがレジストリから取得され、実行中の環境を直接変更することなく置き換えが行われる

### Requirement: イベント駆動ワークロードのサーバーレス実行
イベント駆動・短時間・低頻度のワークロードは、サーバーレス実行基盤（FaaS）で実行されなければならない (SHALL)。

#### Scenario: イベントによる起動
- **WHEN** 対象イベント（キュー投入、ストレージ更新等）が発生したとき
- **THEN** 関数が自動的に起動して処理し、アイドル時のコンピュートコストは発生しない

### Requirement: 水平オートスケーリング
アプリケーションは負荷指標（CPU・メモリ・リクエスト数等）に基づいて自動的に水平スケールしなければならない (MUST)。ノード（基盤側キャパシティ）もワークロード需要に応じて自動増減しなければならない (MUST)。

#### Scenario: 負荷増大時のスケールアウト
- **WHEN** 負荷指標がスケールアウト閾値を超過したとき
- **THEN** レプリカ数が自動的に増加し、必要なら基盤キャパシティも追加される

#### Scenario: 負荷低下時のスケールイン
- **WHEN** 負荷が閾値を下回った状態が継続したとき
- **THEN** レプリカと余剰キャパシティが自動的に縮小され、コストが最適化される

### Requirement: セルフヒーリングと可用性維持
実行基盤は異常なワークロードを自動的に再起動・再配置しなければならない (MUST)。常駐サービスは複数 AZ にレプリカを分散配置しなければならない (MUST)。

#### Scenario: ヘルスチェック失敗時の自動復旧
- **WHEN** ワークロードがヘルスチェックに連続して失敗したとき
- **THEN** 該当インスタンスは自動的に再起動または置き換えられ、トラフィックは健全なレプリカへのみ流れる
